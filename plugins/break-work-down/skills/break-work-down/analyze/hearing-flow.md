# ヒアリング・情報収集フロー詳細

Step 1（自動情報収集）とStep 3（ヒアリング）の詳細手順。

## 第1回 AskUserQuestion（Step 1b: 許可 + 制約条件）

ツール検出直後に、許可取得と制約条件を1回にまとめる。

### 質問の設計

- 質問1 [情報ソース]: 検出済みMCPツール・未接続の優先ツール・ブラウザ履歴を、ユーザーが自分の状況に合わせて選べるようフラットに提示する。特定の選択肢を強調・省略しない
- 質問2 [制約条件]: 提案にあたって考慮すべき制約条件を聞く

未接続ツールを選んだ場合は、`collect/mcp-tools.md` の接続方法を参照して案内する。

---

## ギャップ駆動ヒアリング（Step 3）

### 大前提: 情報が揃っていればヒアリングは不要

自動収集データだけで業務の全体構造・工程分解・ボトルネックの原因・定量化がすべて埋まっている場合、ヒアリングは一切行わずにStep 4に直行する。ヒアリングはギャップを埋めるための手段であり、毎回必ず実施するステップではない。

判定基準: 以下の文を具体的に埋められるなら、ヒアリング不要。
> 「あなたが毎日△△業務の□□工程に◇◇時間費やしている。ボトルネックの原因は○○である。」

### 原則: 「知っていること」は聞かない、「知らないこと」だけ聞く

ヒアリングが必要な場合も、自動収集の結果から何が分かっていて何がまだ不明かを特定し、不明な部分だけをピンポイントで聞く。

「しんどい業務は何ですか？」「時間がかかるのはどれですか？」のような定性的な感想を求める質問は避ける。ブラウザのアクセス頻度やSlack・Telegram・Discordの報告パターンから、時間消費の大きい工程はデータで推定できる。データで推定できないことだけを聞く。

### ギャップの4段階

収集データから、以下の4段階のどこまで把握できているかを判定する。把握できていない段階がヒアリングの対象。

```
段階A: 業務の全体構造
  「この人はどんな業務をしているか」の一覧が分かっているか？
  → 分からない場合: 業務の洗い出しが必要

段階B: 各業務の工程分解
  「その業務はどんなステップで構成されているか」が分かっているか？
  → 分からない場合: 工程の詳細を聞く

段階C: ボトルネックの特定
  「どの工程がボトルネックか」が分かっているか？
  → 分からない場合: 工程ごとの時間配分・原因を聞く

段階D: ボトルネックの定量化
  「そのボトルネックにどれだけ時間がかかっているか」が分かっているか？
  → 分からない場合: 具体的な時間・頻度を聞く
```

収集データの内容から各段階の充足度を判定し、不足している段階のみヒアリングする。

---

### 段階A: 業務の全体構造が不明な場合

**いつ使うか:** 収集データが少なく、業務の一覧自体が推定できない場合。

収集できたデータ（ブラウザ履歴のツール一覧等）を手がかりにして、具体的に聞く。収集データがほぼない場合（例: 店舗マネージャーがスマホ中心で仕事をしている場合）は、「1日の流れ」を聞く方式で業務を洗い出す（「出勤してから最初にやることは？」「午後は主にどんな作業が多い？」「月末・週末に増える作業は？」等）。

**段階Aのゴール:** 業務の一覧（3〜7件）が出揃うこと。業務が多数ある場合（フリーランス等）は、まず大カテゴリ（3〜7件）でグルーピングし、各カテゴリ内の工程分解はStep 2で行う。出揃ったら段階Bに進む。

---

### 段階B: 工程の詳細が不明な場合

**いつ使うか:** 業務名は分かっているが、その業務の中身（サブステップ）が不明な場合。

自動収集で「AI通常編集：1本」「AI欠品編集：1本」と分かっていても、その編集の中で何をしているか（製品情報修正？文章リライト？画像選定？）が見えない場合がこれ。

**データから仮説を立ててから聞く。** ゼロから「何をしていますか？」とは聞かない。

**段階Bの質問設計のルール:**
- 選択肢は収集データから推定した具体的な工程にする（汎用的な選択肢にしない）
- 「○○業務の中で」と業務名を特定した上で聞く（「普段の業務で」のような曖昧な聞き方をしない）
- 1つの業務につき1問で工程を特定し、複数業務がある場合は最も時間消費の大きそうなものから順に聞く

**段階Bのゴール:** 主要業務（1〜3件）の工程が明らかになること。次は段階Cに進む。

---

### 段階C: ボトルネックの原因が不明な場合

**いつ使うか:** 工程は分解できたが、「なぜその工程に時間がかかるのか」の原因が不明な場合。

**段階Cの質問設計のルール:**
- 原因の仮説を選択肢に含める（「AI文章の質が低いから」「ツール間のコピペが面倒だから」等）
- 「なぜ大変ですか？」という開放型の質問ではなく、仮説をぶつける閉鎖型の質問にする
- データから見えている矛盾点やパターンを指摘した上で聞く

**段階Cのゴール:** 主要ボトルネック（1〜3件）の原因が特定できること。次は段階Dに進む。

---

### 段階D: 定量化が不明な場合

**いつ使うか:** ボトルネックとその原因は分かったが、「具体的にどれだけ時間がかかっているか」の数字がない場合。

**段階Dの質問設計のルール:**
- 選択肢はデータから推定した範囲を反映する（まったく見当がつかない数字を並べない）
- 「だいたい」「ざっくり」のトーンで聞く（正確な数字を求めない）
- 複数業務の時間を一度に聞く場合は、業務ごとに分けずに1問で相対比較させる

**段階Dのゴール:** 主要ボトルネックの時間・頻度が数値化できること。これで Step 4（業務削減フェーズ）に進める。

---

## ヒアリングの流れの全体像

```
自動収集完了
  ↓
ギャップの判定: 段階A〜Dのどこまで把握できているか？
  ↓
不足している段階だけヒアリング（段階A → B → C → D の順に進む）
  ↓
「あなたが毎日△△業務の□□工程に◇◇時間費やしている」レベルまで分解できたら終了
  ↓
Step 4（業務削減フェーズ）へ
```

複数の段階を1回のAskUserQuestionにまとめてよい。 例えば段階Bと段階Cの質問を同時に4問まで聞ける。段階は目安であり、ヒアリングの中で複数段階の情報が同時に得られることもある。

## ユーザー特性に応じたヒアリング調整

ユーザーの業種・デバイス・リテラシーに応じて、質問の設計・表現・重点を調整する。

| 特性 | 調整ルール |
|---|---|
| **非デスクワーク業種**（飲食・小売・製造・建設等） | ツール一覧からの業務推定が機能しない。「1日の流れ」を時間帯ベースで聞き、営業時間中と時間外を分けて把握する |
| **Excel/ローカルファイル中心** | ブラウザ履歴に業務が現れない。Excel作業の内訳（どのファイル、どの作業、どの頻度）を明示的に聞く |
| **モバイルアプリ中心** | ブラウザ履歴が空でも「ツールを使っていない」と判断しない。Step 1bで主な業務デバイスを確認し、モバイル主体ならヒアリングで補完 |
| **クリエイティブ職**（デザイナー・ライター等） | 制作作業自体はログに残りにくい。ボトルネックは制作よりも周辺工程（修正やりとり、素材探し等）にあることが多い。制作以外の時間を重点的に聞く |
| **チーム/部門の分析** | 対象者が「自分の業務」と「チームのプロセス」を混同しやすい。段階Aで個人作業とチームフローを明示的に切り分けて聞く |
| **ITリテラシー極低** | 選択肢にツール名や技術用語を含めず、行動で構成する（例: 「手書きのノートに書いている」「スマホにメモしている」等） |

## よくある判断ミス

### 1. ITリテラシーの誤判定

収集データだけでITリテラシーを正確に判定するのは難しい。例えば「kintoneを使っている」だけでは中〜高と判定できない（会社が導入しただけで本人はフォーム入力しかしていない可能性がある）。以下の組み合わせで判定する:
- **高い**: GitHub / API系ツール / n8n・Zapier（自分で設定）/ PowerShell等のスクリプト
- **中**: Notion / Slack / Telegram / Discord / Canva / クラウド会計ソフトを日常利用（ただし設定変更はしない）
- **低い**: Excel + メール中心、紙の帳票が残っている、ブラウザ履歴にクラウドツールが少ない
- **極低**: PCをほぼ使わない、手書き中心、スマホも電話・天気予報・LINE程度。技術用語を一切使わず動作を日常語で説明する。全提案に「導入支援が必要」フラグを付け、段階的導入計画を必須にする
