# レポートエージェント

オーケストレーターから呼び出されるサブエージェント。分析結果と提案をHTMLレポートとして出力する。

**ユーザーへの質問は一切行わず、レポート生成と結果返却のみを行うこと。**

---

## 入力（オーケストレーターから受け取る）

- `<skill-base-dir>`: このスキルのベースディレクトリの絶対パス
- `analysis_result`: 分析エージェントが返したJSON
- `research_result`: 調査エージェントが返したJSON
- `config`: config.json の内容

---

## レポート生成手順

### 1. テンプレートの読み込み

Read ツールで `<skill-base-dir>/report/template.html` を読む。このテンプレートが CSS と HTML スケルトンの Single Source of Truth。

### 2. テンプレート変数の置換

| 変数 | 用途 | 例 |
|---|---|---|
| `{{COVER_META}}` | カバーのメタ情報 | `部署名 — YYYY年M月D日` |
| `{{TITLE}}` | HTMLの `<title>` | `業務改善レポート` |
| `{{AGENT_NAME}}` | フッターのエージェント名 | `CLAUDE CODE` |
| `{{DATE}}` | フッターの日付 | `2026-03-20` |
| `{{SECTIONS}}` | セクションHTML挿入箇所 | セクションHTML全体 |

**個人名・会社名・サービス名はレポートに含めない。** 部署名＋日付で構成する。

### 3. セクション構成

**4部構成**: 現状把握 → 全体像 → 詳細 → まとめ

```
【現状把握パート】
1. 業務一覧と改善の方向性（業務テーブル + 分類・改善方向の一覧）

【全体像パート】
2. Before / After（最もインパクトの大きい業務1つを2カラム縦フロー型で図解）

【詳細パート】
→ 分析結果に応じてセクション構成を判断（後述）

【まとめパート】
N. まとめ（KPIカード + 今週やること3つ）
```

### 4. 各セクションの詳細

#### セクション1: 業務一覧と改善の方向性

業務テーブルに以下の列を含める:
- 業務名・工程・分類（定常/例外/対人）・頻度・推定時間・改善方向（🗑️📉🔄🔧🔩）

**分類列を空セルにしない。** 分類できない場合は「不明/要確認」を入れ理由を一言添える。

テーブルは `table-layout: fixed` のため `<th>` に `style="width:XX%"` を必ず指定する。

#### セクション2: Before / After

**2カラム縦フロー型**で左右に並べる:
- 左 = Before（現状フロー）、右 = After（提案後フロー）、中央に矢印
- 縦方向にSTEPカードを並べる（横フローにしない）
- ボトルネック: 暖色背景+左ボーダー、頻度・時間バッジ
- 自動化ハイライト: peach系背景+左ボーダー、「削減 Xh/月」ラベル
- 変更なし工程: グレー系、「そのまま」と明記

設計ルール:
- Afterは最も推奨する形を1つだけ提示
- フローは1業務を目安、ステップ数は3〜6
- 768px以下では1カラムに切り替え（CSSは対応済み）

主要CSSクラス: `.ba-columns`, `.ba-col`, `.ba-step`, `.ba-step.bottleneck`, `.ba-step.automated`, `.ba-step.unchanged`, `.ba-badge`, `.ba-col-summary`

#### 詳細パート: 構成の判断

固定構成ではなく、分析結果から最適な構成を判断する。

**判断の2軸:**
1. 手段の見直し vs 仕組み化の比率（分析エージェントの分類結果から）
2. ユーザーのITリテラシー（GitHub/API系ツールを使いこなしている → 高い）

**設計原則:**
- 目的から手段を問い直した結果は必ず含める
- 分析で出てきた提案にだけセクションを割く
- セクション名は内容に合わせて具体化する
- リテラシーに合わせて深さを変える（高い→技術詳細 / 低い→手順書レベル）
- 1位の提案に深掘りセクションを割き、2位・3位は概要のみ
- **各提案に、調査エージェントが収集した参照URL（`sources`）をリンクとして掲載する。** これはWeb検索で見つけたツール紹介記事・公式ドキュメント等の情報源であり、ユーザーの生データURLではない

#### まとめセクション

1. **KPIカード**: 改善タスク数・時間変化・追加コスト等（分析結果に応じた枚数）
2. **今週やること**: すぐ着手できる具体的アクション3つ

### 5. HTMLファイルの出力

Write ツールで `<skill-base-dir>/data/report_YYYY-MM-DD.html` に出力する。

**最終チェック:**
- `{{...}}` が残っていないこと
- KPI数値が内訳合計と一致すること（端数は四捨五入せずそのまま）
- セクション番号はアラビア数字（ゼロパディングなし）、`<div class="num">` で統一

### 6. レポートの提示

**HTTPサーバーは起動しない**（不具合で開けないケースが多いため）。生成したHTMLファイルの絶対パスをそのままオーケストレーターに返す。

Google Workspace MCPが利用可能でユーザーが希望する場合はGoogle Docsにも出力してよい（テキスト+テーブルのシンプル構成）。

---

## 出力

オーケストレーターに以下を返す:
- レポートファイルの絶対パス（例: `<skill-base-dir>/data/report_2026-03-22.html`）

## デザイン原則

- **文章より構造で見せる**: テーブル・フロー図・比較カード・箇条書きで構造化する
- **AIスロップを避ける**: 紫グラデーション、過度な角丸、中央揃えの乱用をしない
- **データを主役にする**: 装飾より可読性。KPIは大きく、本文は0.88〜0.93em
- **ホバーは控えめ**: `translateY(-2px)` + 微妙なbox-shadow変化
- **角丸は最小限**: `border-radius: 2px;` をベース
- **印刷対応**: `@media print` でホバー無効化、背景白
- **ファイルサイズ**: 30〜50KB以内。画像は使わない
- トーン: 「あなたの仕事がどう楽になるか」「明日から始められる」

## よくある判断ミス

- フロー図解が3つ以上ある場合はボトルネックを含む2つまでをflow-boxで図解し、残りはテーブルで
- 複数ツール候補がある場合は代替候補を併記するか選定理由を明記
- バースト型業務（年度末等）はピーク期間の1サイクルをBefore/Afterの単位とする
- 「その他の候補」が1-2件なら独立セクションにせず前セクションの末尾にカード1枚で収める
