# AI活用による自動化

ITリテラシーが高く、コストを許容できる現場向けの提案。

## 提案プロセス

1. Web検索で事例・手法を収集する — 対象業務のキーワードで検索し、下記の信頼性の高い専門メディアのドメインを優先的に参照する。ボトルネック業務が複数ある場合は、業務ごとにサブエージェントを割り当てて並行検索する（目安: 2〜3エージェント）
2. 収集した情報をもとに、対象業務に適したAI活用手法・ツールを選定する
3. コスト感と技術詳細を含めて提案としてまとめる
4. 提案の出力時に、参照したWebページのURLを情報源として添付する

## 提案の適合性チェック

AI活用を提案する前に、対象業務・環境との適合性を確認する:

| チェック項目 | 不適合の場合の対応 |
|---|---|
| **既存自動化との重複**: n8n/Zapier等で既に自動化済みか | 新ツール提案ではなく既存ワークフローの拡張（AIノード追加、エラーハンドリング強化等）を優先 |
| **データの機密性**: 規制業種（医療・金融・教育）で外部AIにデータを渡せるか | 不可ならオンプレミスLLMやルールベース自動化に限定 |
| **正確性の要求度**: 経理・法務等で数値・法的判断の正確性が最優先か | 「AIで下書き→人間が検算」フローをベースにし、完全自動化は避ける |
| **クライアント契約**: 受託業務でAI利用が契約上許容されるか | クライアントの品質期待値・AI利用許諾を事前確認 |

## よくある判断ミス

### 1. AI出力品質の変動

「AI議事録生成」等の提案では、生成後に10-20%の修正工数が残ることを見積もりに含める。「人間のレビューが不要になる」という印象を避ける。

### 2. APIコスト予測の精度

月額見積もりは実際の利用パターンで2-3倍になることがある。「最初の1ヶ月は試験運用」という段階的アプローチを明示する。

### 3. AI置き換えが新しい作業を生む

AIが苦手な例外ケースで「AI出力の修正判定」という新タスクが発生する。提案前に「業務を簡素化できないか」を問う。

### 4. リアルタイム性のミスマッチ

顧客対応等のリアルタイム業務にバッチ処理のAI自動化を提案すると、「遅いなら手動でいい」となりやすい。処理遅延と業務の時間的余裕のマッチングを確認する。

### 5. スキルの過信

既存スキルが「できる」と謳っていても、ユーザーの業務に完全にフィットするとは限らない。導入前に小さなテストケースで検証することを提案に含める。

### 6. MCP連携の前提条件

MCPサーバー連携を提案する場合、対象ツールのAPI提供状況とユーザーの環境（管理者権限、ネットワーク制限等）を確認する。
