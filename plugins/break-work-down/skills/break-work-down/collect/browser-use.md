# Browser Use によるWebアプリ収集

MCP連携がないWebアプリ（社内ポータル、業務SaaS等）から、ブラウザ操作でデータを収集する手段。ブラウザ履歴収集（`browser-history.md`）が「どのツールを使っているか」を把握するのに対し、Browser Useは「そのツール上で何をしているか」まで踏み込める。

---

## 位置づけ

| 収集手段 | 把握できること | 前提条件 |
|---|---|---|
| ブラウザ履歴 | どのツールをどれだけ使っているか（ドメイン＋頻度） | なし |
| MCP連携 | ツール内の具体的なデータ（メッセージ、ファイル等） | MCP対応ツールであること |
| Browser Use | MCP非対応ツール内の画面情報（タスク一覧、ステータス等） | Browser Use CLI導入済み＋ユーザーのログイン |

Browser Useは **MCP連携が使えないWebアプリに対する補完手段** という位置づけ。MCP連携が可能なツール（Slack、Notion等）にはMCPを使う。

---

## 前提条件

- Browser Use CLI 2.0がインストール済みであること
- 対象ツールにユーザーがブラウザでログインできること
- **ユーザーの明示的な許可**（どのツールに対してブラウザ操作を行うか、Step 1bで確認）

### 導入確認

```bash
browser-use --version
```

未インストールの場合は案内する:
```bash
pip install browser-use
playwright install chromium
```

導入できない/したくない場合は、ブラウザ履歴収集＋ヒアリングで補完する。Browser Useの導入を強制しない。

---

## 対象ツールの選定

ブラウザ履歴収集の結果から、**MCP非対応かつアクセス頻度が高いWebアプリ**を候補として抽出する。

```
ブラウザ履歴の結果:
  - kintone.com（120回/月）→ MCP未対応 → Browser Use候補 ✓
  - freee.co.jp（80回/月）→ MCP未対応 → Browser Use候補 ✓
  - slack.com（200回/月）→ MCP対応 → MCP連携を使う ✗
  - 社内ポータル（90回/月）→ MCP未対応 → Browser Use候補 ✓
```

候補をユーザーに提示し、どのツールに対してブラウザ操作を許可するか確認する。

---

## 許可取得（Step 1b への追加）

`config.json` に `browser_use` フィールドを追加する:

```json
{
  "browser_use": {
    "enabled": true,
    "allowed_domains": ["kintone.com", "freee.co.jp"],
    "denied_domains": [],
    "login_method": "user_manual"
  }
}
```

### ユーザーへの確認事項

1. **Browser Useでアクセスしてよいツール**（ドメイン単位で許可）
2. **ログイン方法**: ユーザーが手動でログインした状態のブラウザを使う（`user_manual`）
3. **読み取りのみの確認**: Browser Useはデータの読み取りのみ行い、書き込み・変更・送信は一切行わないことを明示する

---

## 収集手順

### Step 1: ログイン状態の確保

Browser Useに認証情報を渡さない。ユーザーに手動ログインしてもらい、そのセッションを引き継ぐ。

```bash
# ユーザーのブラウザプロファイルを使用して起動
# Chrome の場合（Windows）
browser-use --chrome-profile "%LOCALAPPDATA%/Google/Chrome/User Data"
```

ログインセッションが切れている場合は、ユーザーに再ログインを依頼する。自動ログインは行わない。

### Step 2: 画面情報の収集

対象ツールごとに、業務把握に必要な情報だけを取得する。

```
収集する情報（例: プロジェクト管理ツール）:
  - タスク/チケット一覧（タイトル・ステータス・担当者・更新日）
  - ダッシュボード上の集計値（未完了数、進行中数等）
  - ナビゲーション構造（どんなメニュー/機能があるか）

収集する情報（例: 会計・経理ツール）:
  - メニュー構造（どの機能を使っているか）
  - 最近の操作履歴（直近の処理件数・種類）
  - レポート/帳票の種類

収集しない情報:
  - フォームへの入力や送信操作
  - 金額・個人情報などの機微データの詳細値
  - ファイルのダウンロード
```

### Step 3: 収集タスクの記述

Browser Use CLIに渡すタスク記述は、読み取り専用の操作のみに限定する。

```bash
browser-use "kintone.comを開き、アプリ一覧ページに移動して、
各アプリの名前・レコード件数・最終更新日を一覧として取得してください。
データの変更や入力は一切行わないでください。"
```

タスク記述には必ず以下を含める:
- 対象URL（許可されたドメインのみ）
- 取得したい情報の具体的な項目
- 「データの変更・入力・送信は行わない」という制約

---

## 収集結果の整理

Browser Useの出力を、既存の収集データ構造（`data-storage.md` の JSON構造）に統合する。

```json
{
  "sources": ["slack", "browser_chrome", "browser_use_kintone"],
  "tool_environment": [
    { "name": "kintone", "category": "プロジェクト管理", "source": "browser_use",
      "details": "アプリ5個、直近1週間でレコード更新80件" }
  ],
  "estimated_tasks": [
    { "name": "kintoneでの案件ステータス更新", "frequency": "毎日",
      "source": "browser_use_kintone", "details": "案件管理アプリで日次更新、平均15件/日" }
  ]
}
```

`source` フィールドは `browser_use_<ドメイン名>` の形式で記録し、どの手段で取得したかを明示する。

---

## サブエージェントでの並列実行

Browser Use収集は、他の収集サブエージェント（Slack、Notion等）と並列で実行する。

```
許可されたソースが Slack + kintone + freee の場合:

  サブエージェント1: Slack収集（MCP経由）
  サブエージェント2: Browser Use — kintone収集
  サブエージェント3: Browser Use — freee収集

  → 全エージェントの結果を統合してStep 1eへ
```

対象ドメインが複数ある場合は、ドメインごとにサブエージェントを分割する。1つのサブエージェントに複数ドメインをまとめない（ログインセッションやエラーの切り分けのため）。

---

## プライバシーへの配慮

- Browser Useが画面から読み取った情報のうち、業務パターンの推定に必要な項目のみを保存する
- 金額の具体値、個人名、顧客情報などの機微データは要約・匿名化して保存する（例: 「売上レポート3件を処理」→ OK、「A社への請求額120万円」→ NG）
- スクリーンショットは保存しない
- 収集タスク記述に「書き込み・変更・送信禁止」を必ず含め、読み取り専用を徹底する
- 許可されていないドメインへのアクセスは行わない

---

## Browser Use候補から除外すべきケース

対象ツールの選定時（前述の「対象ツールの選定」セクション）に、以下に該当するものはBrowser Use候補から外し、ヒアリングで代替する:

| 除外条件 | 理由 |
|---|---|
| **機密性の高いツール**（士業SaaS、電子カルテ、金融系） | 外部プログラムによるアクセスが利用規約違反・法定守秘義務違反になりうる。監査ログにも痕跡が残る |
| **エンタープライズSaaS**（Salesforce等でIP制限・セッション監査あり） | セキュリティアラート発生やアカウントロックのリスク。事前にIT管理者の確認が必要 |
| **クリエイティブツール**（Figma/Canva等） | 取得できるのはプロジェクト一覧程度で、作業の中身は分からない。ヒアリングの方が効果的 |

また、以下は Browser Use で収集しても把握しきれない業務パターンであり、ヒアリングでの補完が必須:
- **クロスツール転記**: 各ツール単体の画面からはツール間のデータフロー（どのフィールドがどこに転記されるか）が見えない
- **低スペックPC/共有端末**: Playwright + Chromium に約500MBの容量が必要。導入が困難な場合はブラウザ履歴+ヒアリングで代替

## よくある判断ミス

### 1. ログインセッションの有効期限

セッションCookieが短い（30分等）ツールでは、収集中にセッション切れが発生する。長時間の収集が必要な場合は、ツールごとに分割して順次実行する。

### 2. SPA（シングルページアプリ）の画面遷移

React/Vue等のSPAでは、URLが変わらないまま画面が切り替わる。Browser Useのタスク記述では「○○ボタンをクリック」「○○メニューを開く」のようにUI要素で指示する。

### 3. 二要素認証・CAPTCHA

二要素認証やCAPTCHAが表示された場合、Browser Useでは突破できない。ユーザーに手動で認証を完了してもらい、その後の操作をBrowser Useに引き継ぐ。

### 4. UI変更による収集スクリプトの破損

Webアプリの画面構成が変わると、以前のタスク記述が正しく動かなくなる可能性がある。Browser Use 2.0のAIベースの画面認識により従来のRPAより頑健だが、大幅なUIリニューアル時には再調整が必要。収集が失敗した場合は、ブラウザ履歴＋ヒアリングにフォールバックする。

### 5. 操作速度とサーバー負荷

Browser Useの連続操作が対象ツールの利用規約に抵触したり、レート制限に引っかかる可能性がある。操作は最小限にとどめ、一覧ページの情報取得で完結させる。詳細ページへの個別アクセスは必要最小限にする。
