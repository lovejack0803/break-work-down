# MCP連携ツール

Step 1c で許可されたツールごとに実行する収集手順。すべてMCP経由で接続する。

---

## ツール優先度の定義

業務内容の把握に使うツールは優先度で2段階に分類する。

### 優先ツール（業務内容把握向き） — 積極的に収集

| ツール | MCP識別パターン | 業務把握の価値 |
|---|---|---|
| **Slack** | `slack_*` ツール群 | 日常業務・依頼・報告の実態がリアルタイムで記録される |
| **Telegram** | `telegram_*` ツール群 | チーム連絡・タスク指示・報告がチャットに記録される |
| **Discord** | `discord_*` ツール群 | チーム連絡・コミュニティ運営・サポート対応が記録される |
| **Notion** | `notion_*` ツール群 / MCPリソース | 業務フロー・マニュアル・プロジェクト構造が可視化されている |
| **Google Drive** | `gdrive_*` / `google_drive_*` / Google関連リソース | ドキュメント・スプレッドシートに業務プロセスが記録されている |
| **Google Calendar** | `gcal_*` / `google_calendar_*` / Google関連リソース | 定例会議・業務の時間配分パターンが分かる |

### 補助ツール — ユーザーが希望した場合のみ使用

| ツール | MCP識別パターン | 補助扱いの理由 |
|---|---|---|
| n8n / Zapier / Make | `n8n*` / `zapier*` / `make*` | 自動化設定の確認には使えるが、業務の実態把握には向かない |
| GitHub / GitLab | `github*` / `gitlab*` | 開発作業の記録はあるが、業務全体の把握には偏りが生じる |
| その他のMCP | 上記以外 | 業務内容把握との親和性を個別に判断 |

補助ツールはユーザーが希望した場合に使用する。

---

## 接続方法の共通原則

各優先ツールはMCP経由で接続する。MCPが検出済みならそのまま使い、未検出ならセットアップ方法を案内する。「後で接続する」も常に選択肢に含める。

---

## Slack

Slackは業務の実態が最も生々しく記録される場所。

### 接続方法

| 方法 | セットアップ | 読める情報 |
|------|-------------|-----------|
| **A. 公式MCP（推奨）** | `claude plugin install slack` → OAuth認証 | メッセージ検索、チャンネル履歴、ユーザー情報、ファイル |
| **B. 後で接続する** | — | スキップ。ブラウザ履歴等の他ソースで業務を推定 |

**公式 Slack MCP（推奨）**

Slack公式のMCPサーバー（`https://mcp.slack.com/mcp`）。OAuth認証で簡単に接続できる。

セットアップ: `claude plugin install slack` を実行し、ブラウザでOAuth認証するだけ。
**注意**: プラグインインストール後、MCPツールがセッションに反映されるまでClaude Codeの再起動が必要な場合がある。インストール直後にSlackツールが使えない場合は、先にブラウザ履歴等の他ソースで収集を進め、次回セッションでSlack収集を実行する。

## Telegram

チームの業務連絡・タスク指示・進捗報告がTelegramグループやチャンネルに記録されている環境向け。Slackの代わりにTelegramを使っている現場で有効。

### 接続方法

| 方法 | セットアップ | 読める情報 |
|------|-------------|-----------|
| **A. 公式プラグイン（推奨）** | `claude plugin install telegram` → Bot作成・トークン設定 | ボットへのDM・メッセージ送受信 |
| **B. コミュニティMCP（ユーザーボット）** | MTProto対応MCPサーバー設定 → 個人アカウント認証 | 全チャット履歴・グループ・連絡先（個人アカウントとして動作） |
| **C. 後で接続する** | — | スキップ |

**公式 Telegram プラグイン（推奨）**

Claude Code公式のTelegramプラグイン。Botとして動作し、メッセージの受信・返信が可能。

セットアップ:
1. BotFather（@BotFather）でBotを作成し、トークンを取得
2. `claude plugin install telegram` を実行
3. トークンを設定

**コミュニティMCP（ユーザーボット）**

MTProto対応のMCPサーバー（GramJS等）を使い、個人アカウントとして全チャット履歴にアクセスできる。業務把握にはこちらの方が情報量が多い。

**注意**: 個人アカウントのAPI利用はTelegram利用規約に抵触する場合がある。ユーザーに確認してから使用する。

### 収集手順

Slackと同様、仮説駆動で必要な情報だけを取りに行く。

```
① 業務関連のグループ/チャンネル一覧を取得

② メッセージパターンを分析:
   - 定期的な報告（日報・週報）→ 業務サイクルの推定
   - ファイル共有の頻度・種類 → 使用ツール・業務内容の推定
   - タスク指示・完了報告 → 業務フローの推定

収集する情報: グループ名・メッセージ頻度・業務パターンの要約
保存しない情報: メッセージ本文・個人間DM・メディアファイル
```

## Discord

チームの業務連絡やコミュニティ運営、カスタマーサポートがDiscordサーバーで行われている環境向け。

### 接続方法

| 方法 | セットアップ | 読める情報 |
|------|-------------|-----------|
| **A. コミュニティMCP（推奨）** | Discord Bot作成 → MCP設定 | チャンネルメッセージ・サーバー情報 |
| **B. 後で接続する** | — | スキップ |

**コミュニティMCP（推奨）**

Discord MCP サーバーを設定してBotトークンで接続する。メッセージ読み取り・チャンネル一覧取得等が可能。

セットアップ:
1. Discord Developer Portal でBotを作成し、トークンを取得
2. Botをサーバーに招待（Message Content Intentを有効化）
3. MCP設定でBotトークンを環境変数に設定

代表的なMCPサーバー:
- `discord-mcp`（基本的なメッセージ読み書き）
- `discordmcp`（サーバー管理を含む包括的なツール群）

### 収集手順

```
① 業務関連のチャンネル一覧を取得

② チャンネルのメッセージパターンを分析:
   - 定期報告チャンネル → 業務サイクルの推定
   - サポート/問い合わせチャンネル → 対応業務の負荷推定
   - タスク管理Bot（MEE6等）の投稿 → ワークフローの推定

収集する情報: チャンネル名・メッセージ頻度・業務パターンの要約
保存しない情報: メッセージ本文・DM・ファイル添付の中身
```

## Notion

業務フロー・マニュアル・プロジェクト構造がNotionに記録されていることが多い。

### 接続方法

| 方法 | セットアップ | 読める情報 |
|------|-------------|-----------|
| **A. 公式MCP（推奨）** | Integration Token取得 → MCP設定 | ページ、データベース、コメント |
| **B. Claude Code Notionプラグイン** | `claude plugin install notion` | 同上 + スラッシュコマンド |
| **C. 後で接続する** | — | スキップ |

**公式 Notion MCP（`@notionhq/notion-mcp-server`）**

Notion公式のMCPサーバー。Integration Tokenが必要。

セットアップ:
1. https://www.notion.so/profile/integrations → 新しいインテグレーション作成
2. トークン（`ntn_...`）をコピー
3. 対象ページ/データベースをインテグレーションに共有（ページメニュー →「接続を追加」）
4. Claude Code で MCP 設定:
   ```
   claude mcp add notion -- npx -y @notionhq/notion-mcp-server
   ```
   環境変数 `NOTION_TOKEN` にトークンを設定

**Claude Code Notionプラグイン**

`claude plugin install notion` で導入可能。MCP + スラッシュコマンドがセットで使える。OAuth認証対応。

## Google Workspace（Drive + Calendar）

ドキュメント・スプレッドシートの業務プロセスと、カレンダーの会議パターンから業務構造を把握する。

### 接続方法

| 方法 | セットアップ | 読める情報 |
|------|-------------|-----------|
| **A. 公式 gws CLI MCP（推奨）** | Google Cloud OAuth設定 → `gws mcp` 起動 | Drive, Calendar, Sheets, Docs, Gmail 等 |
| **B. コミュニティMCP** | OAuth設定 → MCP設定 | Drive, Calendar（パッケージによる） |
| **C. 後で接続する** | — | スキップ |

**Google 公式 Workspace CLI（`@googleworkspace/cli`）**

Google公式のCLI。Drive, Calendar, Sheets, Docs, Gmail 等を一括カバー。

セットアップ:
1. Google Cloud Console でOAuthデスクトップアプリ認証情報を作成
2. `claude mcp add google-workspace -- npx -y @googleworkspace/cli mcp`
3. 初回実行時にブラウザでOAuth認証

**コミュニティMCP**

`@aaronsb/google-workspace-mcp`（軽量、Drive+Calendar+Gmail）や `google_workspace_mcp`（12サービス対応）など。

### Drive の収集手順

```
① ファイル一覧を取得（最近更新されたもの上位20件）
   → ファイル名・更新日時・種類を確認

② タイトルから業務カテゴリを推定:
   - 「進捗管理」「報告書」「議事録」→ プロジェクト管理系
   - 「マニュアル」「手順書」→ 定型業務の存在を示唆
   - 「集計」「分析」「売上」→ データ集計・レポート系

③ 頻繁に更新されているファイル（週1以上）を3-5件読み込む

収集する情報: ファイル名・更新頻度・種類・内容の要約
保存しない情報: ファイル全文・個人情報・財務データの詳細
```

### Calendar の収集手順

```
① 直近2週間〜1ヶ月のイベント一覧を取得

② 定例パターンを分類:
   - 毎日 → 朝会、報告等
   - 毎週 → 週次MTG、振り返り等
   - 参加人数が多い → 横断的な共有・意思決定

③ 会議の時間帯から推定:
   - 午前に集中 → 午後が作業時間
   - 「準備」「レビュー」が多い → 事前準備コストが高い

収集する情報: タイトル・参加者数・頻度・所要時間
保存しない情報: プライベートな予定の詳細
```

## プライバシーへの配慮

- 外部ツール（Slack・Telegram・Discord等）から取得したメッセージの本文は保存しない。チャンネル/グループ名＋業務パターンの要約のみ
- ユーザーが明示的に削除を依頼した場合は `data/` ごと削除する

## 収集できない環境への対応

自動収集が機能しない主なパターンと、それぞれの代替手段:

| パターン | 原因 | 代替手段 |
|---|---|---|
| **ツールがゼロ**（飲食・小売等） | LINE/紙帳票中心でMCP対応ツールなし | ブラウザ版アプリの履歴で拾う → Browser Use → 完全になければヒアリング直行 |
| **モバイル中心** | スマホアプリが主でPC利用が少ない | Step 1bでデバイスを確認し、モバイル主体ならブラウザ履歴をスキップしヒアリングで補完 |
| **セキュリティ制限**（大企業・公的機関） | OAuth/プラグインが情報ポリシーでブロック | 「技術的に不可（要IT部門承認）」と「ユーザー拒否」を区別。前者はレポートに将来策として記載し、ブラウザ履歴のみに切り替え |
| **法定守秘義務**（士業・医療・金融） | ブラウザ履歴のドメイン情報ですら機密に該当しうる | 自動収集を一切行わず、ユーザーが匿名化した情報のみで分析する選択肢を提示 |

## よくある判断ミス

### 1. ツール検出の過信

ブラウザ版アプリ（Airシフト等）のアクセス履歴で業務パターンを推定できるが、スマホアプリ経由の利用は検出できない。検出結果がゼロ＝業務ツールを使っていない、ではない。
