# 現場目線の超具体的AI活用提案

業務を工程レベルまで分解し、「なくせる業務」を発見した上で、残る業務にシンプルな仕組みを適用するコンサルティングスキル。

## このスキルの軸

AI活用の提案自体はAIエージェントに任せればすぐにできる。しかし現場では「どの業務のどの工程を自動化すべきか」の切り分けができていないため、一般的なAI活用ベストプラクティスを押し付けても成果は出ない。

**このスキルが提供する価値は、業務の構造化・分解と、「なくせる業務」の発見にある。**

DXやAIはあくまで「手段」に過ぎない。**最大のトランスフォーメーションは「業務自体をなくすこと」**であり、ITツールの導入を検討するのは最後。この原則を全ステップで貫く。

- Slackの会話、ブラウザ履歴、ドキュメントから**現場のリアル**を観察する（ただし闇雲に集めず、仮説検証に必要な情報をピンポイントで取りに行く）
- 観察結果から業務を**構造化し、工程単位に分解**する — 業務の入力・処理・出力・判断ポイントを整理し、どこにボトルネックがあるか仮説を立てる
- ヒアリングでは**業務フローの図解やマップ**を提示しながら仮説を検証する — 「○○業務」ではなく「○○業務の△△工程」まで絞り込む
- 分解した工程に対し、まず**「そもそもこれは必要か？」**を問う — 「昔からやっているから」だけで続いている習慣を洗い出し、**廃止・統合・頻度削減**を最優先で提案する
- 廃止できない工程に対してのみ、その現場でしか成立しない**超具体的な仕組み**を提案する — ツールありきではなく、業務に合わせたシンプルな仕組みを構築する

一般論の「AIで業務効率化しましょう」ではなく、「あなたのチームが毎日やっている○○の△△工程、そもそもなくせませんか？　残る□□工程は既に使っている◇◇で自動化すれば××になる」というレベルの具体性を目指す。

## ワークフロー概要

```
【業務分解フェーズ】
  1. 自動情報収集 → 2. 業務の整理・仮説構築 → 3. ヒアリングで仮説検証・ボトルネック特定
【業務削減フェーズ】
  4. 業務の廃止・統合・頻度削減の検討（最大のインパクトはここ）
【解決フェーズ】
  5. 残存業務の分析・優先順位付け → 6. シンプルな仕組みの提案 → 7. リスク評価 → 8. レポート出力
```

スキルの時間配分の大半は**業務分解フェーズ**（Step 1-3）と**業務削減フェーズ**（Step 4）に使ってよい。「なくせる業務」を見つけ出すことが最大の価値であり、残った業務への解決策は自然に導かれる。

**運用原則**:
- ユーザーにドキュメントの準備や業務の列挙を求める前に、まず自動収集する
- 収集した現場データから仮説を立ててからヒアリングに入る（ゼロからの聞き取りはしない）
- 「どの業務をなくせるか」を先に検討し、残った業務に対してのみ解決策を議論する
- ツールありきの提案（プロダクトアウト的発想）を絶対に避ける

## Step 1: 自動情報収集

### 1a. ツール検出

`ListMcpResourcesTool`（server未指定）で全MCPサーバーを検出。さらに、ローカルのブラウザ履歴ファイル（Chrome等）の存在も確認する。

### 1b. 許可取得 + 基本ヒアリング（1回のAskUserQuestionにまとめる）

ツール検出の直後に、**許可・追加ツール提案・基本情報を1回のAskUserQuestionで同時に聞く**（最大4問）。ステップ間の往復を最小化する。

質問の詳細テンプレートは `references/hearing-flow.md` を参照。要点:
- **質問1**: 検出済みツール＋ブラウザ履歴への読み取り許可（セキュリティ注意点を説明）
- **質問2**: 追加で接続したいツールの提案（未接続のツール候補を提示）
- **質問3**: 対象規模（個人/チーム/会社）
- **質問4**: 制約条件（セキュリティ/予算/ITスキル/なし）

検出ツールが0件の場合は質問1を省略し、質問2-4の3問にする。

**許可を得るまで一切の情報読み取りを行わない。**

### 1c. 許可されたツールで情報収集

許可されたツールのみで自動収集を実行する。ツール別の収集手順は `references/tool-collection.md` を参照。

**収集の原則: 仮説駆動で必要な情報だけを取りに行く。**
- まずユーザーの役職・部署・主要チャンネルを把握し、「この人はどんな業務をしているか」の初期仮説を立てる
- その仮説を検証・補強するために必要なチャンネル・検索キーワードを絞って収集する
- 関係の薄いチャンネルやデータまで網羅的に集めない — 時間がかかる上にノイズが増える

追加ツールの接続を希望した場合は、接続方法をテキストでガイドし、設定完了後に収集を実行する。

### 1d. 前回データの読み込み（2回目以降）

`data/` ディレクトリに前回の収集データ（`latest.json`）が存在する場合、読み込んで今回の収集結果と比較する。前回からの変化点（新しいツールの使用開始、業務パターンの変化、チャンネルの増減など）を自動検出し、ヒアリングの起点にする。

前回データがない場合（初回実行）はこのステップをスキップ。

### 1e. 収集結果の整理と保存

収集した情報から「推定業務一覧」「ツール環境」「コミュニケーションパターン」「判明しなかった情報」を整理する。

整理後、収集データを `data/` ディレクトリに保存する。保存手順の詳細は `references/tool-collection.md` の「収集データの保存」を参照。

## Step 2: 業務の整理・仮説構築

Step 1の自動収集結果を業務一覧としてユーザーに提示する。ドキュメントが別途提供された場合は自動収集結果とマージする。

ドキュメント形式別の抽出手順は `references/extraction-patterns.md` を参照。

**ここで重要なのは、業務をフラットに羅列するのではなく、構造化して工程に分解すること。**

収集した現場データ（Slackの会話パターン、ブラウザの利用頻度、ツールの使い分け）を読み解き、業務フローとして構造化する。そこから「この業務の○○工程がボトルネックではないか？」という仮説を構築する。

### 業務の構造化と図解

業務一覧をテキストで提示するだけでなく、**業務フローを図解・マップ化して提示する**。これにより:
- ユーザーが「自分の業務」を俯瞰的に確認でき、抜け漏れに気づきやすい
- 工程間の依存関係やボトルネックが視覚的に明らかになる
- ヒアリング時に「この工程のここですよね？」と具体的に指差せる

```
構造化の例:

■ 業務フローマップ（テキスト図解）

  記事制作フロー:
  [構成作成] → [製品選定] → [AI記事生成] → [人間による編集] → [文章校正] → [予約後チェック] → [公開]
                    ↑                                              ↑
               ボトルネック候補                              多重化の疑い
               (在庫確認が手動)                          (enno.jp + 文賢 + 目視)

■ 仮説構築の例:
- Slackで毎日「受注処理5件」と報告 + ブラウザでCRM 200回・スプレッドシート 80回
  → 仮説:「受注処理の中の転記工程がボトルネック（CRM→スプレッドシートの二重入力）」
- ブラウザで経費精算ツールのアクセスが極めて多い + Slackで「月末の経費まだ？」のリマインド多数
  → 仮説:「経費精算の中の承認工程がボトルネック（申請→差し戻し→再申請のループ）」
```

## Step 3: ヒアリングで仮説検証 — ボトルネックの特定

**このスキルの核心はここにある。** Step 2で立てた仮説を検証し、「どの業務のどの工程がボトルネックか」を確定する。

### 仮説→検証のアプローチ

ヒアリングはゼロからの聞き取りではなく、**仮説の検証**として行う:

1. Step 2で作成した業務フローマップを提示し、「この流れで合っていますか？」と確認する
2. マップ上のボトルネック候補を指して仮説をぶつける（「この工程が大変ではないですか？」）
3. 仮説が合っていれば深掘りする。外れていれば「では実際は？」と修正し、マップを更新する

自動収集の有無で質問内容が変わる:
- **収集ありの場合**: 仮説提示 + 業務確認 + しんどい業務 + 所要時間（検証型）
- **収集なしの場合**: 業務内容 + 困りごと + 所要時間 + ツール環境（聞き取り型）

質問テンプレートの詳細は `references/hearing-flow.md` を参照。

### 業務を分解する3つの視点

AskUserQuestionの回数を減らすことより、**業務を工程レベルまで分解すること**が重要:

1. **「○○業務」を工程に分解して、手動が残っている箇所はどこか？**
   - 「データ入力が面倒」→ 分解すると「収集→入力→チェック→転記」の4工程。ボトルネックは「転記」（二重入力）
   - 「議事録作成が面倒」→ 分解すると「録音→文字起こし→要約→アクション抽出→共有」。ボトルネックは「要約→アクション抽出」
   - 業務名ではなく、その中の具体的な工程を特定する

2. **ユーザーが言っていないが時間を食っている工程はないか？**
   - 「当たり前すぎて挙げていない」工程にこそ改善余地がある
   - ブラウザ履歴で高頻度アクセスなのにヒアリングで触れられないツール → 無意識の時間消費

3. **人数 × 時間の掛け算で最もインパクトが大きい工程はどこか？**
   - 時間削減だけでなく、品質向上・ミス防止・精神的負荷の軽減も含めて考える
   - 「7人×30分/日」の工程は「1人×3時間/日」の工程よりチーム全体への波及効果が大きい

### ボトルネックが特定できるまで解決策に入らない

業務の分解が十分でなければ追加のAskUserQuestionを柔軟に使う。「もう少し聞いてもいいですかね？」のトーンで。**「○○チームが毎日△△業務の□□工程に◇◇時間費やしている」レベルまで分解できて初めて、Step 4に進む。**

### ヒアリング時に「そもそもこれ必要？」の視点を持つ

ボトルネックを特定する過程で、以下の問いを常に意識する（Step 4で本格検討するための伏線）:
- この業務は「昔からやっているから」続いているだけではないか？
- この報告・チェック・承認は本当に毎回必要か？
- 別の工程と統合すれば丸ごとなくせないか？

ただしこの段階では無理に結論を出さず、「なくせるかもしれない候補」としてメモしておく。

**自動収集で既に分かっていることは聞かない。**

## Step 4: 業務の廃止・統合・頻度削減の検討（業務削減フェーズ）

**ここが提案の質を根本から変える最重要ステップ。** Step 3で特定したボトルネックに対し、ITツールを検討する前に「**その業務自体をなくせないか？**」を徹底的に問う。

### 「なくす」が最大のDX

多くの提案者が「今の業務をITで楽にする」と考えがちだが、本当のDXは**「その業務自体をなくす」**ことにある。業務をなくすことができれば、それが最もインパクトの大きいトランスフォーメーション。導入コストゼロ、運用コストゼロ、学習コストゼロ。

### 3つの削減アプローチ

各ボトルネック業務に対し、以下の順序で検討する（上ほど優先）:

1. **廃止**: そもそもこの業務・工程は必要か？
   - 「昔からやっている」だけの慣習ではないか？
   - 誰も見ていない報告書、形骸化した承認フロー、重複するチェック工程
   - 例: 「日次報告 → 本当に毎日必要？週次で十分では？」「全員承認 → 代表者承認で十分では？」

2. **統合**: 別の業務・工程とまとめられないか？
   - 同じデータを複数箇所に入力している → 一箇所に統合
   - 複数の会議で同じ情報を共有している → 1つの場で完結
   - 例: 「CRMとスプレッドシートの二重入力 → そもそもスプレッドシート側が不要では？」

3. **頻度削減**: 回数・頻度・対象を減らせないか？
   - 毎日 → 週次、全件 → 抽出チェック、全員 → 担当者のみ
   - 例: 「全案件の品質チェック → 一定金額以上のみ」「毎朝の進捗共有 → ツールで可視化して必要時のみ」

### 合意形成のコミュニケーション

業務をなくす提案は現場の抵抗を受けやすい。以下のアプローチで合意形成を促す:
- **データで語る**: 「この報告書、過去3ヶ月でSlackで言及されたのは2回だけです」
- **問いかけで導く**: 「この工程がなくなったら、何か困ることはありますか？」
- **段階的に**: 「まず2週間やめてみて、問題がなければ正式に廃止しませんか？」

### このステップの出力

各ボトルネック業務を以下に分類する:
- **🗑️ 廃止候補**: なくしても問題ない業務・工程（根拠付き）
- **🔗 統合候補**: 別の業務とまとめられる業務・工程
- **📉 頻度削減候補**: 回数・範囲を減らせる業務・工程
- **🔧 仕組み化候補**: 上記で対処できず、ITツールの適用が必要な業務・工程

**🔧に分類された業務のみがStep 5に進む。** 廃止・統合・頻度削減だけで解決するものはレポートで大きくハイライトする（これが最大の成果になることが多い）。

## Step 5: 残存業務の分析・優先順位付け

Step 4で「🔧 仕組み化候補」に分類された業務を**時間消費が多い順にトップ5**を特定し、そこから**改善候補トップ3**を選定する。

選定基準: 時間削減インパクト × 導入しやすさ × ユーザーのしんどさ

分析の詳細フレームワークは `references/analysis-framework.md` を参照。

## Step 6: シンプルな仕組みの提案

**ツールありきのプロダクトアウト的発想を避ける。** 業務を整理・削減した上で、どうしても残る工程に対してのみ、シンプルな仕組みを適用する。

### 提案の原則

- **既存ツールを最大活用**: 新しいツールを導入する前に、既に使っているツールで解決できないか検討する
- **シンプルさを最優先**: Google フォーム、スプレッドシート、AppSheetのような汎用ツールの組み合わせで十分なことがほとんど
- **現場が迷わず使える仕組み**: 以下の3条件を満たすこと
  - 一度データを入れたら二度手間にならない
  - リアルタイムで可視化される
  - メンテナンス（手直し）が不要

### 提案の構成

**1位に絞って具体的な作業フローを設計する**。2位・3位は概要のみ。

- 作業フロー: ユーザーがやること → 仕組みが返すもの
- 始め方の3ステップ（所要時間付き）
- ツール選定は環境・制約を最優先（`references/tool-catalog.md` 参照）
- AI/高度なツールは、シンプルな仕組みでは対応できない場合にのみ提案する

## Step 7: リスク評価

提案ツールについてWebSearchで最新情報を確認。レポート内では独立セクションではなく統合的に反映:
- **個人**: つまずきやすいポイント内
- **チーム**: 運用ルール例内
- **会社**: リスクと対策セクション

調査手順は `references/security-evaluation.md` を参照。

## Step 8: レポート出力

### 出力フォーマット一覧

| フォーマット | 必須/任意 | 用途 | ガイド |
|---|---|---|---|
| **HTML** | 必須 | メインレポート（読む資料） | `references/report-html.md` |
| **Excel** | 任意 | ROI試算・業務分析データ（触る資料） | `references/report-xlsx.md` |
| **PPTX** | 任意 | プレゼン用スライド | `references/report-pptx.md` |
| **PDF** | 任意 | HTMLから変換（印刷・メール添付用） | `references/report-pdf.md` |

HTMLは常に生成する。他のフォーマットは以下の条件で**提案**する:
- **Excel**: 会社パターン、または「数字をいじりたい」「上に報告する」と言われた場合
- **PPTX**: 「プレゼンする」「スライドが欲しい」と言われた場合
- **PDF**: 「メールで送りたい」「印刷したい」「稟議添付」と言われた場合

### 7a. HTMLレポート（必須）

**単一の静的HTMLファイル**（React不使用）。`references/report-html.md` のカラーパレット・CSS仕様に従う。

**構成**: カバー → 3分まとめ(KPI) → 業務トップ5 → **業務削減提案（廃止・統合・頻度削減）** → 仕組み化候補トップ3 → 1位深掘り → ツール → つまずきポイント → パターン別追加セクション → Next Action

**業務削減提案セクションを目立たせる**: 廃止・統合・頻度削減の提案は「ツール不要で最大インパクト」としてレポートの前半でハイライトする。これが最大の成果になることが多い。

### 7b. Excelワークブック（任意）

`references/report-xlsx.md` に従う。ROI試算シートは**計算式付き**で、ユーザーが数値を変更するとシミュレーション結果が自動更新される設計にする。

### 7c. PPTXスライド（任意）

`references/report-pptx.md` に従う。

### 7d. PDF（任意）

`references/report-pdf.md` に従う。HTMLレポートから変換するため、追加のデザイン作業は不要。

## 重要な注意点

- **提案は現場で観察した具体的事実に基づくこと** — Slackの会話やブラウザ履歴から見えた実際の業務パターンに根ざした提案を行う。一般的なAI活用ベストプラクティスの押し付けではなく、その現場でしか成立しない超具体的な提案にする
- **「なくせないか？」を常に最初に問う** — すべてのボトルネックに対し、ITツールの適用を考える前に「この業務自体をなくせないか」「統合できないか」「頻度を減らせないか」を検討する。ツール導入は最後の手段
- **ツールありきの提案をしない** — 「このAIツールが便利だから使いましょう」ではなく「この業務課題をどう解決するか」から逆算する。Google フォーム+スプレッドシートで十分な場合にAI SaaSを提案しない
- **ボトルネックの特定と業務削減の検討が終わるまで解決策に入らない** — Step 3で「○○業務の△△工程」レベルまで分解し、Step 4で廃止・統合・頻度削減を検討して初めてStep 5に進む
- **ユーザーにドキュメントの準備を求めない** — まずStep 1で自動収集を試みる
- **外部ツールへのアクセスは必ずユーザーの許可を得てから** — 勝手に読み取らない
- **自動収集で分かったことは聞き直さない**
- **AskUserQuestionは効率よく使うが、業務の分解と削減検討に必要なら何度でも聞く** — 許可+基本情報はまとめて聞いて効率化する一方、「どの業務をなくせるか、どの工程に仕組みが必要か」の特定には十分な対話を行う。回数制限より分解の粒度を優先する
- ツールをハードコードしない — ListMcpResourcesToolで動的検出
- 制約条件をツール選定に直接反映する
- 会社パターンではPoC → Go/No-Go判定を必ず含める
- ITリテラシーが低い環境では具体的対策を明記する（アンバサダー制度、印刷ガイド、1対1サポート）
