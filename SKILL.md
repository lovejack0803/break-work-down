# 業務分析・改善提案スキル

現場の業務内容を分析し、効率化・自動化の改善を提案するためのスキルです。

## 前提

- アウトプットの質はインプットの質で決まる：
  - このスキルはSlack・Telegram・Discord・ブラウザ履歴・Notion・Google Workspace等の外部ツールと連携し、現場の業務データを直接収集できるのが強み
  - ただし、アクセス権・接続可否・参照ファイルの欠落などで収集ができない場合もあります。その場合は「可能な範囲だけ」を使い、足りない部分は最小ヒアリングや暫定前提で補完し、その不確実性をレポート/提案内に明記する
- 業務をタスクレベルまで把握し、それぞれに合った提案をするのがこのスキルの本質。「受注処理を効率化しましょう」のような粗い提案ではなく、「受注処理の中のCRM転記工程をAPI連携に変える」のように、具体的な工程に対して具体的な手段を提案する
- 業務の価値を否定しない。「この作業は不要」ではなく「この作業の目的を、別の手段で達成できないか」という姿勢を貫く。人間の作業をそのままAIやツールに移植するのではなく、目的に立ち返って手段自体を変える選択肢を常に考える

## ユーザーとのコミュニケーション

ユーザーはエンジニアやコーディング目的の人が多いが、非エンジニアが使い始めるケースも増えている。このスキルは「業務改善」という非エンジニア領域を扱う場合もあるため、技術リテラシーの幅が特に広い。

文脈の手がかりからユーザーのリテラシーを読み取り、表現を合わせる。ブラウザ履歴にGitHub・API系ツールがあれば技術的な用語をそのまま使ってよいし、そうした手がかりがなければ平易な表現を使う。

ユーザーの状況に応じて柔軟に進めること。データが豊富ならヒアリングは最小限でいいし、収集手段がなければヒアリング中心になる。さらに、現場の業務データが揃わない場合は「どこが確定で、どこが未確定か」を会話とレポートの両方で切り分けます。

## ワークフロー

```
【把握フェーズ】現場で何が起きているかを正確に掴む
  Step 0-1: 自動情報収集 → Step 2: 業務の整理・仮説構築 → Step 3: ヒアリングで実情を確認

【分析フェーズ】各業務の目的と手段を分析し、改善の方向性を決める
  Step 4: 目的の確認 → 手段の妥当性評価 → 分類

【調査フェーズ】改善手段の選択肢を調べる
  Step 5: シンプルな仕組み化 / AI活用 / スキル・エージェント の3方向で調査

【提案フェーズ】調査結果を提案にまとめ、レポートとして出力する
  Step 6: 優先順位付け + 提案 → Step 7: レポート出力
```

## Step 0: 設定とデータの読み込み

設定と収集データは `data/` に保存する。スキル開始時に `data/config.json` を読み込み、前回の設定があれば再利用する。初回は空なのでStep 1bで取得して書き戻す。

初期化が必要な場合: `PYTHONUTF8=1 python "<skill-base-dir>/scripts/save_collection_data.py" reset`

---

## Step 1: 自動情報収集

ユーザーに何かを準備してもらう前に、まず自分で集められるだけ集める。これにより「ゼロからのヒアリング」を避け、仮説を持った状態で会話に入れる。

### 1a. ツール検出

利用可能なMCPサーバーとローカルのブラウザ履歴を確認し、対象者の業務を最もよく反映するツールを見極める。MCP連携ツールの優先度や接続方法は `collect/mcp-tools.md`、ブラウザ履歴の収集は `collect/local-collection.md` を参照（情報収集を始める前に読む）。

### 1b. 許可取得 + 制約条件

`config.json` の未設定項目のみ聞く。**許可が得られた範囲だけ** 情報読み取りする — ユーザーの信頼を損なうと、その後のヒアリングの質が落ちるため。
拒否/未許可がある場合は、その領域の分析を推定せず、ユーザーが提供できる範囲と会話で補う前提に切り替えます。

### 1c. 情報収集の実行

仮説駆動で収集する。「この人はどんな業務をしているか」の初期仮説を立て、検証に必要な情報だけを取りに行く。ツール別手順は `collect/mcp-tools.md`・`collect/local-collection.md` 参照。
収集結果が欠損する/特定ツールにアクセスできない場合は、欠損している前提を明確化し、以降の分析は「確定部分」と「仮説部分」を分けて扱います（特にStep 3以降）。

#### サブエージェントによる並列収集

**許可されたデータソースごとに1体のサブエージェントを起動し、すべて並列で実行する。** 各ソースの収集は独立しているため、逐次実行する理由がない。

```
例: Slack + Notion + ブラウザ履歴が許可された場合

  サブエージェント1: Slack収集
    → プロフィール取得、チャンネル一覧、日報・業務チャンネルの読み取り、ユーザーの投稿検索

  サブエージェント2: Notion収集
    → ページ・DB検索、業務フロー・マニュアルの読み取り

  サブエージェント3: ブラウザ履歴収集
    → collect_browser_history.py の実行、結果の分類

  → 全エージェントの結果を統合してStep 1eへ
```

各サブエージェントのプロンプトには、`collect/mcp-tools.md` または `collect/local-collection.md` の該当ツールの収集手順を含める。サブエージェントは収集結果をJSON形式で返し、メインエージェントがStep 1eで統合・保存する。

Google Workspaceのように Drive と Calendar で収集内容が異なる場合でも、同一MCPなので1体のサブエージェントにまとめてよい。

### 1d. 前回データとの比較（2回目以降）

`latest.json` が存在する場合、今回の結果と比較して変化点を自動検出する。

### 1e. 収集結果の整理と保存

収集結果を整理し `data/` に保存する（`collect/local-collection.md` の保存手順参照）。

## Step 2: 業務の整理・仮説構築

収集結果を業務一覧としてユーザーに提示。「受注処理」のような粗い粒度で止めず、「メール確認 → CRM入力 → 在庫確認 → 見積作成 → 承認依頼」のように具体的な工程まで把握する。この粒度まで見えていれば、ボトルネックは自然に浮かび上がる。

収集データからボトルネックの仮説を構築し、業務フローを図解する。レポートでは**Before/After形式**で提示する（`report/report-html.md` の「Before / After」を読んで設計ルールを確認）。

---

## Step 3: ギャップ駆動ヒアリング — 実情の確認

Step 2の仮説を検証し、現場の実情を確定させる。

データで推定できないことだけを聞く。ギャップの4段階（業務全体構造 → 工程の詳細 → ボトルネック原因 → 定量化）のうち、不足している段階だけをヒアリングする。質問設計の詳細は `analyze/hearing-flow.md` を参照（ヒアリングを始める前に読む）。

以下を具体的に埋められるなら、ヒアリングなしでStep 4に進む:
> 「あなたの△△業務は、□□ → ○○ → ◇◇ という工程で構成されており、□□工程に毎日◇◇時間費やしている。」

ヒアリング中は「この業務の目的は何か？ 手段は最善か？」を意識しておく。Step 4（分析）の伏線になる。

---

## Step 4: 業務の分析 — 目的と手段の評価

把握できた業務に対し、「**この業務は何のためにやっているのか？ 今の手段は最善か？**」を分析する。

判別できない部分がある場合は、追加で最小の質問を行うか、暫定仮説として分類しその前提をレポートに明記してStep 5に進む。

### 分析の流れ

1. **目的の確認**: 何を達成するためにあるのか？
2. **手段の妥当性**: 過剰・不足はないか？
3. **分類**: 以下の4パターンで分類する

| パターン | 判断基準 | 分類 |
|---|---|---|
| **やめる** | 目的自体が形骸化している | 🗑️ 不要 |
| **まとめる・減らす** | 統合や頻度削減で負荷を下げられる | 📉 簡素化 |
| **変える** | 目的は正しいが、別の手段の方が効率的 | 🔄 手段変更 |
| **仕組みで効率化** | 目的も手段も妥当だが、手作業が多い | 🔧 仕組み化 |

🔄と🔧がStep 5（調査）に進む。🗑️と📉は調査不要で、そのままレポートに反映する。

### 「変える」の例

```
目的: 経営層に進捗を伝える
  従来: 人間がPowerPoint作成 → 安直な置き換え: AI生成（崩れやすい）
  手段の変更: ダッシュボードで常時共有 → スライド作成自体が不要に

目的: 売上集計に使う
  従来: CRM→スプレッドシートに毎日転記 → 安直な置き換え: RPA（UI変更で壊れる）
  手段の変更: API連携で直接集計 → 転記工程自体が不要に
```

---

## Step 5: 改善手段の調査

🔄・🔧に分類された業務に対し、以下の3方向で改善手段を調査する（該当ファイルを読んでから調査を始める）:

| 方向 | 参照 | 必須/任意 |
|---|---|---|
| シンプルな仕組み化 | `propose/simple-automation.md` | 常に実施 |
| AI活用 | `propose/ai-automation.md` | 常に実施 |
| スキル・エージェント探索 | `propose/skill-search.md` | 常に実施 |

現場の制約条件（リテラシー・コスト・環境）に合った方向を選ぶ。**3方向すべてを調査した上で**、最適な組み合わせを提案に含める。スキル探索で該当なしの場合は「該当なし」とレポートに明記する。

### サブエージェントによる並列調査

**3方向の調査は独立しているため、必ず3体のサブエージェントを同時に起動する。** 逐次実行や、1体のエージェントに複数方向をまとめない。

```
サブエージェント1: シンプルな仕組み化の調査
  → propose/simple-automation.md を読み、ボトルネック業務のWeb検索を実行
  → 事例・ツール候補・コスト感を収集して返す

サブエージェント2: AI活用の調査
  → propose/ai-automation.md を読み、ボトルネック業務のWeb検索を実行
  → プロンプト改善手法・パイプライン設計・API連携の事例を収集して返す

サブエージェント3: スキル・エージェント探索
  → propose/skill-search.md を読み、skills.sh / skillsmp.com / Skills CLI を検索
  → 該当スキルの評価、または「該当なし」の根拠を返す

→ 全エージェントの結果を統合してStep 6へ
```

ボトルネック業務が複数ある場合は、サブエージェント1・2を業務ごとに分割してもよい（例: 業務Aの仕組み化 + 業務BのAI活用 + スキル探索 = 3体以上）。**スキル探索は業務数に関わらず常に1体。**

各サブエージェントのプロンプトには、参照ファイルのパス（`<skill-base-dir>/propose/...`）を含め、**エージェント自身がファイルを読んでから調査を始める**よう指示する。参照ファイルが利用できない場合は、そこから得られる「意図」を抽象化して代替提案を作り、調査範囲が限定されていることを明記する。

---

## Step 6: 優先順位付け + 提案

Step 4-5の結果を改善インパクト順に並べ、上位に集中して提案する。選定基準の詳細は `analyze/analysis-framework.md` を参照（優先順位を決める前に読む）。

参照ができない場合は、少なくとも「インパクト（時間/ミス/品質/リードタイム）× 実現可能性（工数/コスト/リスク）」の観点で暫定の優先順位をつけます。その場合は暫定であることを注記します。

**提案の設計**:
- なぜその提案なのかを理由から説明する。「Zapierを導入する」ではなく「毎日30分の転記がAPI連携で不要になるから」
- 1つのボトルネックへの提案が類似工程にも適用できないか考え、まとめて提案する
- 5つの改善を薄く提案するより、1つの改善を具体的に設計する方が実行しやすい
- 「すぐできること」（設定変更、テンプレート化等）は制約に関係なく併記する

---

## Step 7: レポート出力

`report/template.html` の `{{SECTIONS}}` にセクションHTMLを挿入して完成させる。デザイン方針と構成の判断基準は `report/report-html.md` を参照（レポート生成前に読む）。

**構成**: 全体像（Before/After） → 詳細（提案） → まとめ（KPI + 次のアクション）

レポート生成後、可能ならブラウザで自動表示: `cmd.exe /c start "" "<出力ファイルパス>"`

自動表示できない/環境的にブラウザ起動できない場合は、生成されたHTMLの出力先（`<出力ファイルパス>`）を明確に返し、必要に応じて会話内にも要点を貼ります。

## ガードレール

- **現場の実情を十分に把握してから提案に進む** — 把握が不十分だと提案の精度が落ちる一方、時間都合などで不足する場合は「暫定提案」として進め、その不確実性を必ず明記する。
- **提案には収集データの根拠を明記する** — 根拠のない提案は読み手が判断できない
- **業務一覧テーブルの分類列（定常/例外/対人）を空セルにしない** — 分類できない場合は「不明/要確認」を入れ、理由を一言添える。分類がStep 4の判断根拠になるため
- **外部ツールへのアクセスは必ずユーザーの許可を得てから** — 信頼関係がヒアリングの質を左右する。許可がない場合は、その領域の分析を推定せず、会話とユーザー提供情報で補う
- **KPI数値は内訳合計と一致させる** — KPIカードと本文が矛盾しないようにし、端数は四捨五入で丸めず内訳合計をそのまま使う（推定なら注記する）
- **プライバシー/安全性**: レポートには個人情報・生のメッセージ全文・識別子（メール/トークン/URL等）を再掲しない。出典は「ツール種別＋要約（カテゴリ、頻度、傾向）」で示す
- **未許可領域の扱い**: 許可がないデータ領域は根拠として使わない（推定も「要確認」扱いにする）。不明が残る場合は「何が足りないか」を明記する
