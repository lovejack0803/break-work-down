# 現場目線の業務削減提案

## このスキルの4ステップ

1. **現場の面倒を集める** — Slack・ブラウザ履歴等から、何に時間を取られているかデータで把握する（→ Step 0〜1）
2. **手順をすべて書き出してボトルネックを探す** — 業務を工程単位に分解し、どこが詰まっているか特定する（→ Step 2〜3）
3. **そもそもやめられないか考える** — 廃止・統合・頻度削減を最優先で検討する。ITツールの検討はこの後（→ Step 4）
4. **残った仕事にだけ仕組みを入れる** — どうしても残る工程に対してのみ、シンプルなツールやAIを適用する（→ Step 5〜7）

## ワークフロー概要

```
【業務分解フェーズ】
  1. 自動情報収集 → 2. 業務の整理・仮説構築 → 3. ヒアリングで仮説検証・ボトルネック特定
【業務削減フェーズ】
  4. 業務の廃止・統合・頻度削減の検討
【解決フェーズ】
  5. 残存業務の分析・優先順位付け → 6. シンプルな仕組みの提案 → 7. レポート出力
```

スキルの時間配分の大半は**業務分解フェーズ**（Step 1-3）と**業務削減フェーズ**（Step 4）に使ってよい。「なくせる業務」を見つけ出すことが最大の価値であり、残った業務への解決策は自然に導かれる。

**運用原則**:
- ユーザーにドキュメントの準備や業務の列挙を求める前に、まず自動収集する
- 収集した現場データから仮説を立ててからヒアリングに入る（ゼロからの聞き取りはしない）
- 「どの業務をなくせるか」を先に検討し、残った業務に対してのみ解決策を議論する
- ツールありきの提案（プロダクトアウト的発想）を絶対に避ける

## Step 0: 設定とデータの読み込み

設定と収集データはすべて `data/` ディレクトリに保存する。

```
data/
  ├── config.json                  ← ユーザー設定（制約条件、優先チャンネル等）
  ├── latest.json                  ← 最新の収集データ（クイックアクセス用）
  └── collection_YYYY-MM-DD.json   ← 日付入りスナップショット
```

スキル開始時に `data/config.json` を読み込む。設定が空の場合（初回実行）はStep 1bで取得した情報を書き戻す。2回目以降は保存済みの設定を再利用し、質問を省略する。

```json
{
  "slack_priority_channels": [],
  "constraints": [],
  "browser": "auto",
  "output_dir": ""
}
```

**設定の更新:** Step 1bの回答をconfig.jsonに書き戻す。次回以降は設定済みの項目をスキップし、未設定の項目だけ聞く。

**初期化:** 前回の収集データをクリアしてゼロから始めたい場合は以下を実行する:
```bash
PYTHONUTF8=1 python "<skill-base-dir>/scripts/save_collection_data.py" reset
```
config.jsonが初期状態に戻り、collection_*.json / latest.json が全削除される。

## Step 1: 自動情報収集

### 1a. ツール検出 + 優先度分類

`ListMcpResourcesTool`（server未指定）で全MCPサーバーを検出。さらに、ローカルのブラウザ履歴ファイル（Chrome等）の存在も確認する。`config.json` の `browser` 設定があればそれに従う。

業務内容把握に直結するツールを優先する。対象者の職種や業務内容によって優先度は変わる。エンジニアならGitHub、自動化担当ならn8n等が最優先になりうる。

検出したツールの中から、対象者の業務を最もよく反映しているものを見極め、Step 1b で許可を求めて積極的に収集する（`collect/tool-collection.md` の優先度定義を参照）。

### 1b. 許可取得 + 接続方法の選択 + 制約条件

`config.json` に `constraints` が設定済みなら制約条件の質問をスキップする。`slack_priority_channels` が設定済みならSlackの収集対象チャンネルとして使う。

未設定の項目がある場合のみAskUserQuestionで聞く:
- **情報ソースの選択**: 検出済みツール（★印）と未接続の優先ツール候補を1つの質問にまとめる
- **制約条件**（未設定の場合のみ）: セキュリティ/予算/ITスキル等

回答を得たら `config.json` に書き戻す。

未接続ツールをユーザーが選んだ場合は、接続方法を `collect/tool-collection.md` に従ってテキストで案内する。各ツールにはMCP・直接API・プラグイン等の複数の接続手段があり、ユーザーの環境に合った方法を提示する。「後で接続する」も常に許容する。

**許可を得るまで一切の情報読み取りを行わない。**

### 1c. 許可されたツールで情報収集

許可されたツールのみで自動収集を実行する。ツール別の収集手順は `collect/tool-collection.md` を参照。

収集の優先順位: **MCP → `scripts/` 配下の専用スクリプト → インラインコード**。専用スクリプトにはエンコーディング対応（Windows cp932問題の回避）、ロック回避、プライバシーフィルタリングが組み込まれている。インラインコードを使う場合はこれらに自分で対応すること。

**接続方法のフォールバック**: 優先ツールはMCP経由を最初に試し、MCPが使えない場合はスクリプトによる直接API接続にフォールバックする（Slack等）。MCPの有無だけで収集を諦めない。

**収集の原則: 仮説駆動で必要な情報だけを取りに行く。**
- まずユーザーの役職・部署・主要チャンネルを把握し、「この人はどんな業務をしているか」の初期仮説を立てる
- その仮説を検証・補強するために必要なチャンネル・検索キーワードを絞って収集する
- 関係の薄いチャンネルやデータまで網羅的に集めない — 時間がかかる上にノイズが増える

追加ツールの接続を希望した場合は、接続方法をテキストでガイドし、設定完了後に収集を実行する。

### 1d. 前回データの読み込み（2回目以降）

`data/` に前回の収集データ（`latest.json`）が存在する場合、読み込んで今回の収集結果と比較する。前回からの変化点（新しいツールの使用開始、業務パターンの変化、チャンネルの増減など）を自動検出し、ヒアリングの起点にする。

前回データがない場合（初回実行）はこのステップをスキップ。

### 1e. 収集結果の整理と保存

収集した情報から「推定業務一覧」「ツール環境」「コミュニケーションパターン」「判明しなかった情報」を整理する。

整理後、収集データを `data/` に保存する。保存手順の詳細は `collect/tool-collection.md` の「収集データの保存」を参照。

## Step 2: 業務の整理・仮説構築

Step 1の自動収集結果を業務一覧としてユーザーに提示する。ドキュメントが別途提供された場合は自動収集結果とマージする。

ドキュメント形式別の抽出手順は `collect/extraction-patterns.md` を参照。

**ここで重要なのは、業務をフラットに羅列するのではなく、構造化して工程に分解すること。**

収集した現場データ（Slackの会話パターン、ブラウザの利用頻度、ツールの使い分け）を読み解き、業務フローとして構造化する。そこから「この業務の○○工程がボトルネックではないか？」という仮説を構築する。

### 業務の構造化と図解

業務一覧をテキストで提示するだけでなく、**業務フローを図解・マップ化して提示する**。

```
構造化の例:

■ 業務フローマップ（テキスト図解）

  記事制作フロー:
  [構成作成] → [製品選定] → [AI記事生成] → [人間による編集] → [文章校正] → [予約後チェック] → [公開]
                    ↑                                              ↑
               ボトルネック候補                              多重化の疑い
               (在庫確認が手動)                          (enno.jp + 文賢 + 目視)

■ 仮説構築の例:
- Slackで毎日「受注処理5件」と報告 + ブラウザでCRM 200回・スプレッドシート 80回
  → 仮説:「受注処理の中の転記工程がボトルネック（CRM→スプレッドシートの二重入力）」
- ブラウザで経費精算ツールのアクセスが極めて多い + Slackで「月末の経費まだ？」のリマインド多数
  → 仮説:「経費精算の中の承認工程がボトルネック（申請→差し戻し→再申請のループ）」
```

## Step 3: ギャップ駆動ヒアリング — ボトルネックの特定

**このスキルの核心はここにある。** Step 2で立てた仮説を検証し、「どの業務のどの工程がボトルネックか」を確定する。

### 原則: 知っていることは聞かない、知らないことだけ聞く

「しんどい業務は？」「困っていることは？」のような定性的な感想は求めない。ブラウザのアクセス頻度やSlackの報告パターンから、時間消費の大きい工程はデータで推定できる。**データで推定できないことだけを聞く。**

### ギャップの4段階

収集データから、以下の4段階のどこまで把握できているかを判定し、不足している段階だけヒアリングする:

```
段階A: 業務の全体構造が分かっているか？ → 不明なら業務の洗い出し
段階B: 各業務の工程分解ができているか？ → 不明なら工程の詳細を聞く
段階C: ボトルネックの原因が分かっているか？ → 不明なら原因の仮説をぶつける
段階D: ボトルネックが定量化できているか？ → 不明なら時間・頻度を聞く
```

自動収集の充実度によって開始段階が変わる:
- **Slack + ブラウザ + ドキュメント**: 段階B〜Cから開始（業務一覧は推定済み）
- **ブラウザのみ / Slackのみ**: 段階A〜Bから開始
- **収集データなし**: 段階Aから開始

各段階の質問設計の詳細は `analyze/hearing-flow.md` を参照。

### 質問設計のルール

- **選択肢は収集データから推定した具体的な工程にする**（汎用的な選択肢にしない）
- **仮説をぶつける閉鎖型の質問にする**（「なぜ大変ですか？」ではなく「○○が原因ですか？△△が原因ですか？」）
- **1つの業務につき1問で工程を特定する**（複数業務がある場合は時間消費の大きそうなものから）

### ヒアリングなしでStep 4に進めるケース

自動収集データだけで段階A〜Dのすべてが埋まっている場合、**ヒアリングは行わずにStep 4に直行する**。ヒアリングはギャップを埋めるための手段であり、目的ではない。

判定基準: 以下の文を具体的に埋められるなら、ヒアリング不要。
> 「あなたが毎日△△業務の□□工程に◇◇時間費やしている。ボトルネックの原因は○○である。」

埋められない部分がある場合のみ、その部分だけをピンポイントでヒアリングする。

### ヒアリング時に「そもそもこれ必要？」の視点を持つ

ボトルネックを特定する過程で、以下の問いを常に意識する（Step 4で本格検討するための伏線）:
- この業務は「昔からやっているから」続いているだけではないか？
- この報告・チェック・承認は本当に毎回必要か？
- 別の工程と統合すれば丸ごとなくせないか？

ただしこの段階では無理に結論を出さず、「なくせるかもしれない候補」としてメモしておく。

## Step 4: 業務の廃止・統合・頻度削減の検討（業務削減フェーズ）

**ここが提案の質を根本から変える最重要ステップ。** Step 3で特定したボトルネックに対し、ITツールを検討する前に「**その業務自体をなくせないか？**」を徹底的に問う。

### 3つの削減アプローチ

各ボトルネック業務に対し、以下の順序で検討する（上ほど優先）:

1. **廃止**: そもそもこの業務・工程は必要か？
2. **統合**: 別の業務・工程とまとめられないか？
3. **頻度削減**: 回数・頻度・対象を減らせないか？

### このステップの出力

各ボトルネック業務を以下に分類する:
- **🗑️ 廃止候補**: なくしても問題ない業務・工程（根拠付き）
- **🔗 統合候補**: 別の業務とまとめられる業務・工程
- **📉 頻度削減候補**: 回数・範囲を減らせる業務・工程
- **🔧 仕組み化候補**: 上記で対処できず、ITツールの適用が必要な業務・工程

**🔧に分類された業務のみがStep 5に進む。** 廃止・統合・頻度削減だけで解決するものはレポートで大きくハイライトする（これが最大の成果になることが多い）。

---

## Step 5: 残存業務の分析・優先順位付け

Step 4で「🔧 仕組み化候補」に分類された業務を**時間削減インパクトの大きい順に優先順位をつけ、上位に集中して提案する**。

選定基準: 時間削減インパクト × 導入しやすさ × ユーザーのしんどさ

分析の詳細フレームワークは `analyze/analysis-framework.md` を参照。

## Step 6: シンプルな仕組みの提案

**ツールありきのプロダクトアウト的発想を避ける。** 業務を整理・削減した上で、どうしても残る工程に対してのみ、シンプルな仕組みを適用する。

### 提案の原則

- **既存環境と新しい選択肢をフラットに比較**: 既存ツールの拡張も新規導入も、現場に合うかどうかで判断する。「○○を使っているから△△」という短絡的な提案をしない。ツール選定時は必ずWebSearchで最新の選択肢を確認する
- **シンプルさを最優先**: 汎用ツールの組み合わせで十分なことがほとんど

### 提案の構成

**最もインパクトの大きい提案に注力して具体設計する**。他の提案は優先度に応じて詳細度を調整する。

- 作業フロー: ユーザーがやること → 仕組みが返すもの
- 始め方の3ステップ（所要時間付き）
- ツール選定は環境・制約を最優先（WebSearchで最新情報を確認）
- AI/高度なツールは、シンプルな仕組みでは対応できない場合にのみ提案する
- **ツールのレッドフラグ確認**: WebSearchでツール名を検索する際、重大インシデント・サービス終了・明らかな悪評がないか併せて確認する。問題が見つかった場合はレポートの注意点セクションに事実のみ記載する。見つからなければ「導入前に自社のセキュリティ基準で確認を推奨」と一文添える

## Step 7: レポート出力

### 出力フォーマット

| フォーマット | 必須/任意 | 用途 | ガイド |
|---|---|---|---|
| **HTML** | 必須 | メインレポート（読む資料） | `report/report-html.md` |
| **Excel (.xlsx)** | 任意 | 業務一覧・削減効果の試算（触る資料） | `report/report-xlsx.md` |

HTMLは常に生成する。Excelは必要と判断した場合に提案する（Step 7b参照）。

### 7a. HTMLレポート（必須）

`report/template.html` をベースに使用する。テンプレートにはCSS・カバー・フッターの定型部分が含まれている。`{{SECTIONS}}` プレースホルダーにセクションのHTMLを挿入して完成させる。デザイン方針は `report/report-html.md` を参照。

**構成は「全体像 → 詳細 → まとめ」の3部構成。** 全体像パートとまとめパートは共通。**詳細パートはパターンによって変化する。**

**詳細パートの構成は固定ではない。** 分析結果（削減 vs 仕組み化の比率）とユーザーのITリテラシーに応じて、セクションの数・名前・深さをその場で判断する。判断の2軸と設計原則は `report/report-html.md` の「詳細パートの構成方針」を参照。

**業務の全体像を最初に提示**: 業務フローマップとボトルネック箇所を最初に見せ、読み手が全体の文脈を掴んでから詳細に入る構成にする。KPIや次のアクションは最後のまとめに集約する。

**レポート生成後、ブラウザで自動表示する**:
```bash
# Windows環境での日本語ファイル名対応
cmd.exe /c start "" "<出力ファイルパス>"
```

### 7b. Excel (.xlsx) の提案

Excel出力を提案する。ユーザーが必要と答えた場合のみ生成する。

`report/excel_helpers.py` の `ReportWorkbook` クラスを使用する。カラーパレット・スタイル・ヘルパー関数が組み込まれているので、Claudeはデータの流し込みに集中すればよい。

```
提案の例:
「続いて、Excelファイル（.xlsx）も作成しますか？」
```

ユーザーが希望した場合、`report/report-xlsx.md` に従い `openpyxl` で生成する。HTMLレポートと情報量を一致させること。

## ガードレール

- **外部ツールへのアクセスは必ずユーザーの許可を得てから** — 勝手に読み取らない
- **業務の構造を理解し、削減の余地を検討してからツール提案に進むこと**
- ツールをハードコードしない — ListMcpResourcesToolで動的検出し、制約条件をツール選定に直接反映する
- **自律的に行動する** — 次のステップに進める状態なら確認質問を挟まず進む。AskUserQuestionはユーザーの意思決定が本当に必要な場面（情報ソースの選択、ボトルネックの検証等）のみに使う。「気になる点はありますか？」のような不要な確認は行わない
